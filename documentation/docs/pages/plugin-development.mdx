# Plugin Development

This guide covers how to create plugins for Boo — both external JavaScript plugins and built-in Swift plugins.

## External Plugins

External plugins live in `~/.boo/plugins/` and are hot-loaded at runtime. No restart required — save a file and it appears in the sidebar.

Plugins are JavaScript, running in-process via JavaScriptCore — no dependencies, no shell, instant execution.

### Directory Structure

```
~/.boo/plugins/my-plugin/
  plugin.json       # Required manifest
  main.js           # Plugin logic
```

### Manifest (`plugin.json`)

```json
{
  "id": "my-plugin",
  "name": "My Plugin",
  "version": "1.0.0",
  "icon": "star",
  "description": "A custom sidebar plugin",
  "runtime": "js",
  "capabilities": { "sidebarPanel": true }
}
```

That's all you need to get started. The full set of fields:

#### Manifest Fields

| Field | Required | Description |
|---|---|---|
| `id` | Yes | Unique plugin identifier |
| `name` | Yes | Display name in sidebar and status bar |
| `version` | Yes | Semver version string |
| `icon` | Yes | [SF Symbol](https://developer.apple.com/sf-symbols/) name |
| `description` | No | Human-readable description |
| `when` | No | When-clause expression for visibility (null = always visible) |
| `runtime` | No | `"js"` (only supported value) |
| `capabilities.sidebarPanel` | No | Show in sidebar |
| `capabilities.statusBarSegment` | No | Show in status bar |
| `statusBar.position` | No | `"left"` or `"right"` |
| `statusBar.priority` | No | Lower = closer to edge |
| `statusBar.template` | No | Template string for status bar text |
| `settings` | No | Array of setting declarations (see [Plugin Settings](#plugin-settings)) |

#### Plugin Settings

Plugins can declare their own settings, which appear in the Settings → Plugins UI under an expandable section. Each setting in the `settings` array has:

| Field | Required | Description |
|---|---|---|
| `key` | Yes | Setting key (unique within this plugin) |
| `type` | Yes | `"bool"`, `"string"`, `"int"`, or `"double"` |
| `label` | Yes | Human-readable label shown in UI |
| `default` | No | Default value |
| `options` | No | UI hint: `"fontPicker:system"` for system font picker, `"fontPicker:mono"` for monospace font picker |

Example manifest with settings:

```json
{
  "id": "my-plugin",
  "name": "My Plugin",
  "version": "1.0.0",
  "icon": "star",
  "runtime": "js",
  "capabilities": { "sidebarPanel": true },
  "settings": [
    { "key": "enabled", "type": "bool", "label": "Enable feature", "default": true },
    { "key": "fontSize", "type": "double", "label": "Font size", "default": 12.0 },
    { "key": "fontName", "type": "string", "label": "Font", "default": "", "options": "fontPicker:system" }
  ]
}
```

Settings are stored per-plugin in `AppSettings` and accessed via:

```swift
AppSettings.shared.pluginBool("my-plugin", "enabled", default: true)
AppSettings.shared.pluginDouble("my-plugin", "fontSize", default: 12.0)
AppSettings.shared.pluginString("my-plugin", "fontName", default: "")
AppSettings.shared.setPluginSetting("my-plugin", "fontSize", 14.0)
```

#### Status Bar Templates

Templates substitute values from the terminal context:

| Token | Value |
|---|---|
| `{cwd}` | Current working directory |
| `{process.name}` | Foreground process name |
| `{git.branch}` | Current git branch |
| `{git.changedCount}` | Number of changed files |
| `{remote.host}` | Remote host or container name |

### Writing a Plugin

Provide `main.js` with a `render` function. It receives the terminal context and returns DSL elements. Runs in JavaScriptCore — no process spawn, no dependencies.

#### Minimal example

```javascript
function render(ctx) {
  var dir = ctx.cwd.split("/").pop();
  return {
    type: "vstack",
    children: [
      { type: "label", text: "Hello from " + dir, style: "bold" },
      { type: "button", label: "Open in Finder", action: { type: "reveal", path: ctx.cwd } }
    ]
  };
}
```

Both `render` and `transform` are accepted as the entry function name. You can return an object or call `JSON.stringify()` — both work.

#### Context object (`ctx`)

| Field | Type | Example |
|---|---|---|
| `ctx.cwd` | string | `"/Users/phlp/project"` |
| `ctx.processName` | string | `"node"` |
| `ctx.isRemote` | bool | `false` |
| `ctx.envType` | string | `"local"`, `"ssh"`, `"docker"` |
| `ctx.remoteHost` | string | `"server.example.com"` |
| `ctx.paneCount` | number | `2` |
| `ctx.tabCount` | number | `3` |
| `ctx.git.branch` | string | `"main"` |
| `ctx.git.repoRoot` | string | `"/Users/phlp/project"` |
| `ctx.git.isDirty` | bool | `true` |
| `ctx.git.changedFileCount` | number | `5` |
| `ctx.enrichedData` | object | Data from other plugins |
| `ctx.settings` | object | Plugin settings declared in manifest (see [Plugin Settings](#plugin-settings)) |

#### Host functions

JS plugins have access to these functions provided by the host:

| Function | Returns | Description |
|---|---|---|
| `readFile(path)` | string or null | Read a file relative to `ctx.cwd`. Returns null if not found. |
| `fileExists(path)` | bool | Check if a file exists relative to `ctx.cwd`. |

Absolute paths are allowed but must be under `ctx.cwd` (security boundary).

#### Example: reading a project file

```javascript
function render(ctx) {
  var raw = readFile("package.json");
  if (!raw) return { type: "label", text: "No package.json", style: "muted" };

  var pkg = JSON.parse(raw);
  var scripts = pkg.scripts || {};
  var names = Object.keys(scripts);
  if (!names.length) return { type: "label", text: "No scripts defined", style: "muted" };

  return {
    type: "list",
    items: names.map(function (name) {
      return {
        label: name,
        icon: "play.circle",
        detail: scripts[name],
        action: { type: "exec", command: "npm run " + name }
      };
    })
  };
}
```

### DSL Elements

Your `render` function returns a JSON object or array of elements. Each element has a `type` and type-specific properties:

| Type | Properties | Description |
|---|---|---|
| `label` | `text`, `style?`, `tint?` | Text label. Styles: `"bold"`, `"muted"`, `"mono"`. Alias: `text` (with `content` field) |
| `button` | `label`, `action`, `style?`, `contextMenu?` | Clickable button. Styles: `"primary"`, `"secondary"`, `"destructive"` |
| `divider` | — | Horizontal line |
| `spacer` | — | Vertical spacing |
| `icon` | `name`, `tint?` | SF Symbol icon. Tints: `"success"`, `"error"`, `"warning"`, `"accent"`, `"muted"` |
| `list` | `items` | List of items with `label`, `detail?`, `icon?`, `action?`, `contextMenu?` |
| `vstack` | `children` | Vertical layout of child elements |
| `hstack` | `children` | Horizontal layout of child elements. Alias: `row` |
| `badge` | `text`, `tint?` | Colored pill badge |

### DSL Actions

Actions are attached to interactive elements (buttons, list items):

| Action Type | Parameters | Effect |
|---|---|---|
| `"cd"` | `path` | Changes terminal directory |
| `"open"` | `path` | Opens file in `$EDITOR` |
| `"exec"` | `command` | Sends command to terminal |
| `"copy"` | `text` or `path` | Copies to clipboard |
| `"reveal"` | `path` | Opens in Finder |

### Context Menus

List items and buttons can have a `contextMenu` array for right-click actions:

```javascript
{
  label: "main.js",
  icon: "doc",
  action: { type: "open", path: "main.js" },
  contextMenu: [
    { label: "Open in editor", icon: "pencil", action: { type: "open", path: "main.js" } },
    { label: "Copy path", icon: "doc.on.doc", action: { type: "copy", text: "main.js" } },
    { label: "Delete", style: "destructive", action: { type: "exec", command: "rm main.js" } }
  ]
}
```

Each menu item has `label` (required), `action` (required), and optional `icon` (SF Symbol) and `style` (`"destructive"` for red text).

### When-Clauses

Control when your plugin is visible based on terminal state:

| Expression | Meaning |
|---|---|
| `null` | Always visible |
| `"git.active"` | Only in git repos |
| `"!remote"` | Only in local sessions |
| `"remote"` | Only in remote sessions |
| `"remote.type == 'ssh'"` | Only in SSH sessions |
| `"remote.type == 'docker'"` | Only in Docker sessions |
| `"git.active && !remote"` | Git repos, local only |
| `"process.name == 'node'"` | Only when Node.js is running |
| `"process.editor"` | Only when an editor is active |
| `"process.category == 'database'"` | Only for database clients |

Process categories: `editor`, `vcs`, `runtime`, `build`, `package`, `container`, `database`, `monitor`, `filemanager`, `multiplexer`.

Operators: `&&` (and), `||` (or), `!` (not), `==`, `!=`, `()` (grouping).

See [When-Clauses](/plugins#when-clauses) for the full list of variables.

### Example: Makefile targets plugin

A JS plugin that reads a Makefile and lists its targets as clickable items:

**`~/.boo/plugins/make-targets/plugin.json`**

```json
{
  "id": "make-targets",
  "name": "Make Targets",
  "version": "1.0.0",
  "icon": "hammer",
  "description": "Lists Makefile targets. Click to run.",
  "when": "env.local",
  "runtime": "js",
  "capabilities": { "sidebarPanel": true }
}
```

**`~/.boo/plugins/make-targets/main.js`**

```javascript
function render(ctx) {
  var src = readFile("Makefile");
  if (!src) return { type: "label", text: "No Makefile", style: "muted" };

  var targets = [];
  var lines = src.split("\n");
  for (var i = 0; i < lines.length; i++) {
    var m = lines[i].match(/^([a-zA-Z][a-zA-Z0-9_-]*)\s*:/);
    if (m) targets.push(m[1]);
  }
  if (!targets.length) return { type: "label", text: "No targets found", style: "muted" };

  return {
    type: "list",
    items: targets.map(function (t) {
      return { label: t, icon: "terminal", action: { type: "exec", command: "make " + t } };
    })
  };
}
```

All example plugins are in the `examples/plugins/` directory:

| Plugin | What it demonstrates |
|---|---|
| **hello-world** | Minimal starting point — `plugin.json` + 12 lines of JS |

## Built-in Swift Plugins

Built-in plugins are Swift classes compiled into the app. They have full access to AppKit, SwiftUI, and all Boo APIs.

### File Organization

All code for a plugin lives in one directory:

```
Boo/Plugins/YourPlugin/
  YourPlugin.swift          # Plugin class (BooPluginProtocol)
  YourPluginDetailView.swift  # SwiftUI detail view (optional)
  YourService.swift          # Supporting service (optional)
```

Tests go in the matching directory:

```
Tests/BooTests/Plugins/YourPlugin/
  YourPluginTests.swift
```

No `Package.swift` changes needed — SPM resolves recursively within the target directories.

### Reference Example: SystemInfoPlugin

The `SystemInfoPlugin` (`Boo/Plugins/SystemInfo/SystemInfoPlugin.swift`) is a working example that demonstrates all plugin patterns:

- **Manifest** with `when: "!remote"` (hidden in SSH/Docker sessions)
- **Status bar** via `makeStatusBarContent` showing memory usage with color-coded tint
- **Sidebar panel** via `makeDetailView` with SwiftUI progress bars and action buttons
- **Dynamic section title** via `sectionTitle` showing live memory percentage
- **Background refresh** via a 10-second timer calling `onRequestCycleRerun`
- **Lifecycle callback** (`cwdChanged`) to refresh disk usage on directory change
- **Host actions** via `DSLActionHandler` to run `df -h` and `top` in the terminal
- **Tests** in `Tests/BooTests/Plugins/SystemInfo/SystemInfoPluginTests.swift`

### Minimal Plugin

```swift
import SwiftUI

@MainActor
final class MyPlugin: BooPluginProtocol {
    var hostActions: PluginHostActions?
    var onRequestCycleRerun: (() -> Void)?

    let manifest = PluginManifest(
        id: "my-plugin",       // Also used as pluginID (derived automatically)
        name: "My Plugin",
        version: "1.0.0",
        icon: "star",
        description: "Example built-in plugin",
        when: nil,             // nil = always visible; or use "git.active", "!remote", etc.
        runtime: nil,
        capabilities: PluginManifest.Capabilities(sidebarPanel: true, statusBarSegment: true),
        statusBar: PluginManifest.StatusBarManifest(position: "right", priority: 25, template: nil),
        settings: [
            PluginManifest.SettingManifest(key: "greeting", type: .string, label: "Greeting text", defaultValue: AnyCodableValue("Hello"), options: nil)
        ]
    )

    // MARK: - Status Bar

    func makeStatusBarContent(context: TerminalContext) -> StatusBarContent? {
        StatusBarContent(
            text: "Hello",
            icon: "star",
            tint: nil,
            accessibilityLabel: "My Plugin"
        )
    }

    // MARK: - Detail View

    func makeDetailView(context: TerminalContext, actionHandler: DSLActionHandler) -> AnyView? {
        AnyView(Text("Hello from My Plugin")
            .foregroundColor(Color(nsColor: AppSettings.shared.theme.chromeText)))
    }
}
```

### Registration

Add your plugin to `PluginRegistry.registerBuiltins()`:

```swift
func registerBuiltins() {
    register(FileTreePlugin())
    register(GitPlugin())
    register(DockerPluginNew())
    register(BookmarksPluginNew())
    register(SystemInfoPlugin())
    register(MyPlugin())  // Add here
}
```

### Using Host Actions

Access terminal and window actions through the injected `hostActions`:

```swift
// Paste a path into the active terminal
hostActions?.pastePathToActivePane?("/some/path")

// Open a directory in a new tab
hostActions?.openDirectoryInNewTab?("/some/dir")

// Open a directory in a new split pane
hostActions?.openDirectoryInNewPane?("/some/dir")

// Send raw text to the active terminal
hostActions?.sendRawToActivePane?("ls -la\r")

// Access the terminal bridge for advanced integration
if let bridge = hostActions?.bridge {
    // Subscribe to bridge events, read terminal state, etc.
}
```

### Triggering Refreshes

When your plugin's data changes asynchronously (e.g., from a background process, file watcher, or network callback), call `onRequestCycleRerun?()`:

```swift
// In your async callback
SomeService.shared.onDataChanged = { [weak self] in
    DispatchQueue.main.async {
        self?.onRequestCycleRerun?()
    }
}
```

This triggers a full plugin cycle, which rebuilds the sidebar and status bar.

### Lifecycle Callbacks

Override lifecycle methods to respond to terminal state changes:

```swift
// Terminal changed working directory
func cwdChanged(newPath: String, context: TerminalContext) {
    refreshData(for: newPath)
}

// SSH/Docker session started or ended
func remoteSessionChanged(session: RemoteSessionType?, context: TerminalContext) {
    if session != nil {
        // Entering remote session
    } else {
        // Back to local
    }
}

// Focus moved to a different pane
func terminalFocusChanged(terminalID: UUID, context: TerminalContext) {
    // Update cached state for the new pane
}
```

### Enrichment

If your plugin contributes data that other plugins need, implement `enrich()`:

```swift
func enrich(context: EnrichmentContext) {
    // Write data visible to all plugins in Phase 2
    context.setData(myValue, forKey: "my-plugin.myKey")
}
```

Other plugins read this in their `react()` phase via the frozen `TerminalContext`:

```swift
func react(context: TerminalContext) {
    if let value = context.enrichedData["my-plugin.myKey"] {
        // Use the value contributed by the other plugin
    }
}
```

JavaScript plugins can access enriched data via `context.enrichedData["key"]`.

### Conditional Visibility

Set the `when` field in your manifest to control when your plugin appears. The `whenClause` property is automatically derived from `manifest.when` — no override needed:

```swift
let manifest = PluginManifest(
    id: "my-plugin",
    ...
    when: "git.active && !remote",  // Only visible in local git repos
    ...
)
// whenClause is automatically parsed from manifest.when
```

### Scroll Behavior

By default, the sidebar wraps your detail view in an `NSScrollView`. If your view handles its own scrolling (like `FileTreeView` does), return `false`:

```swift
var prefersOuterScrollView: Bool { false }
```

### Dynamic Section Titles

Override `sectionTitle(context:)` to show context-dependent headers:

```swift
func sectionTitle(context: TerminalContext) -> String? {
    let dirName = (context.cwd as NSString).lastPathComponent
    return dirName.isEmpty ? nil : dirName
}
```

Return `nil` to use the static `manifest.name`.

## Status Bar Segments

For adding standalone segments to the status bar (not tied to a sidebar plugin), use the `StatusBarPlugin` protocol:

```swift
final class MySegment: StatusBarPlugin {
    let id = "my-segment"
    let position: StatusBarPosition = .left
    let priority = 25

    func isVisible(settings: AppSettings, state: StatusBarState) -> Bool { true }

    func draw(at x: CGFloat, y: CGFloat, theme: TerminalTheme,
              settings: AppSettings, state: StatusBarState, ctx: CGContext) -> CGFloat {
        let attrs: [NSAttributedString.Key: Any] = [
            .font: NSFont.systemFont(ofSize: 10),
            .foregroundColor: theme.chromeMuted
        ]
        let str = "Hello" as NSString
        str.draw(at: NSPoint(x: x, y: y), withAttributes: attrs)
        return str.size(withAttributes: attrs).width
    }

    func handleClick(at point: NSPoint, in barView: StatusBarView) -> Bool { false }
    func update(state: StatusBarState) {}
}
```

Register in `StatusBarView.registerDefaultPlugins()`.

Sidebar plugins **do not** need to create their own status bar segments — the registry auto-registers `PluginIconSegment` toggle icons for any plugin with `capabilities.sidebarPanel = true`.
